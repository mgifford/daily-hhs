# Definition of Done: Daily HHS A11y Scan Report

> A published daily report is considered **done** when all of the following criteria are met.

This document applies to every daily report generated by the `Daily HHS A11y Scan` pipeline and published to GitHub Pages.

---

## 1. Content Complete

The report includes all required categories and data:

- Aggregate Lighthouse scores for all five categories: Performance, Accessibility, Best Practices, SEO, and PWA.
- List of top HHS URLs scanned in the run, with per-URL scan outcomes.
- Traffic-weighted accessibility impact estimates by disability group (Section 508 FPC categories).
- Constrained-network slow-risk indicators for mobile and rural page-load contexts.
- URL counts: processed, succeeded, failed, and excluded.

**Verified by:**
- `src/publish/build-daily-report.js` - assembles the daily report payload
- `src/aggregation/impact-estimation.js` - computes impacted-user estimates
- `src/aggregation/score-aggregation.js` - aggregates Lighthouse scores
- `kitty-specs/002-daily-dap-quality-benchmarking/contracts/daily-report.schema.json` - schema contract

---

## 2. Data Integrity Verified

The published report payload is schema-valid and internally consistent:

- Daily report JSON validates against the contract schema without errors.
- Trend and history values are drawn from the same pipeline outputs, not recalculated independently.
- URL counts are consistent (processed = succeeded + failed + excluded).
- Prevalence profile and severity weight values are loaded from `src/config/prevalence.yaml`; invalid or incomplete values cause the run to fail fast rather than publish misleading data.

**Verified by:**
- `tests/contract/report-schema.test.js` - schema contract validation
- `tests/unit/build-daily-report.test.js` - payload builder unit tests
- `tests/unit/impact-estimation.test.js` - impact calculation accuracy
- `tests/unit/config-validation.test.js` - config integrity checks

---

## 3. Publication Complete

The report is published to the expected public paths and the dashboard is updated:

- Date-stamped snapshot written to `docs/reports/daily/YYYY-MM-DD/report.json` and `docs/reports/daily/YYYY-MM-DD/index.html`.
- Top-level dashboard page updated at `docs/reports/index.html`.
- History index updated at `docs/reports/history.json`.

**Verified by:**
- `src/publish/archive-writer.js` - writes date-stamped snapshots
- `src/publish/build-history-index.js` - updates history index
- `src/publish/render-pages.js` - generates HTML pages
- `.github/workflows/daily-scan.yml` - commits and pushes `docs/reports/` snapshots

---

## 4. Archive and Navigation Complete

The report is reachable via the history index and all navigation links resolve:

- The new report entry appears in `docs/reports/history.json` within the configured lookback window (default: 31 days; see `src/config/prevalence.yaml`).
- Links from the dashboard to daily report pages and back are valid.
- Archive index page lists the new entry alongside prior reports within the lookback range.

**Verified by:**
- `src/publish/build-history-index.js` - lookback-window-filtered index
- `tests/unit/build-history-index.test.js` - history generation and windowing tests

---

## 5. Accessible by Design

The generated HTML meets WCAG 2.2 AA standards:

- All rendered HTML pages include `lang="en"`, `charset="utf-8"`, and a `viewport` meta tag.
- Semantic HTML is used throughout: `<main>`, `<header>`, `<footer>`, `<table>` with proper `<th>` headers, correct heading hierarchy.
- No `aria-label` is placed on elements that lack a valid ARIA role.
- Automated axe-core WCAG 2.2 AA checks pass on every generated page type:
  - Daily report page
  - Dashboard page
  - Code quality page
  - Archive index page
  - 404 page
  - Failure page

**Verified by:**
- `tests/unit/axe-html-accessibility.test.js` - axe-core WCAG 2.2 AA checks on all render functions
- `.github/workflows/scan-github-pages.yml` - monthly axe scan of published Pages site
- `ACCESSIBILITY.md` - project accessibility commitment and guidelines

---

## 6. Secure Rendering Confirmed

All external or user-controlled values are safely escaped before rendering in HTML:

- Every value sourced from scan results, DAP data, or configuration that appears in HTML output is passed through `escapeHtml()`.
- No raw URL, page title, or scan result is inserted directly into HTML without escaping.
- External links include `rel="noreferrer"` to prevent referrer leakage.

**Verified by:**
- `src/publish/render-pages.js` - `escapeHtml()` applied to all dynamic values
- `src/publish/failure-report.js` - `escapeHtml()` applied to error details
- `AGENTS.md` and `ACCESSIBILITY.md` - escaping rules enforced for all contributors and AI agents

---

## 7. Failure Behavior Covered

If a run fails at any stage, an explicit failure report is published rather than leaving the output stale or empty:

- A failure report JSON and HTML page are written to `docs/reports/daily/YYYY-MM-DD/` with diagnostic details (error message, run date, run ID).
- The failure is uploaded as a GitHub Actions artifact for operator review.
- The pipeline returns a non-zero exit code to signal the failure to CI.
- No partial or misleading report is published when a fatal error occurs.

**Verified by:**
- `src/publish/failure-report.js` - `buildFailureReport()`, `renderFailurePage()`, `writeFailureSnapshot()`
- `tests/unit/failure-report.test.js` - failure report unit tests
- `.github/workflows/daily-scan.yml` - failure-report publication step

---

## 8. Quality Gate Passed

All automated tests pass before a report run is considered production-ready:

```
npm test
```

The test suite covers:
- Unit tests for all aggregation, publication, and scanning modules
- Contract tests validating report JSON against the schema
- Integration smoke tests for the end-to-end pipeline
- Accessibility tests (axe-core WCAG 2.2 AA) on all generated HTML pages

**Verified by:**
- `tests/unit/` - unit tests for all modules
- `tests/contract/report-schema.test.js` - schema contract validation
- `tests/integration/daily-scan-smoke.test.js` - end-to-end smoke tests
- `.github/workflows/ci.yml` - CI pipeline runs `npm test` on every pull request

---

## 9. Stakeholder-Ready

The published report is understandable by a general public audience, not just technical readers:

- Each accessibility finding includes the affected disability groups via Section 508 FPC icons and human-readable labels.
- Impacted-user estimates include clear caveats identifying them as rough population-level estimates, not precise measurements.
- Lighthouse score categories are labeled with plain-language descriptions.
- Trendlines are presented with the lookback window clearly stated.
- A Call to Action section links to resources such as Section 508, Open Scans, and USWDS adoption guidance.
- The data source (analytics.usa.gov/health-human-services, powered by DAP) is attributed prominently.

**Verified by:**
- `src/publish/render-pages.js` - labels, caveats, trend window, and CTA sections
- `README.md` - disability FPC icon table with population estimates and source citations
- `src/config/prevalence.yaml` - data reviewed annually; next review date noted inline

---

## Summary Checklist

Use this checklist to assess any individual daily report before treating it as complete:

- [ ] **Content complete**: All five Lighthouse categories, top URLs, and impact estimates are present.
- [ ] **Data integrity verified**: Report JSON is schema-valid and figures are internally consistent.
- [ ] **Publication complete**: Date-stamped snapshot, dashboard, and history index are all updated.
- [ ] **Archive/navigation complete**: Report is reachable via history within the configured lookback window.
- [ ] **Accessible by design**: Generated HTML passes WCAG 2.2 AA axe-core checks.
- [ ] **Secure rendering confirmed**: All external data is HTML-escaped; no unsafe output.
- [ ] **Failure behavior covered**: Run published either a completed report or an explicit failure report.
- [ ] **Quality gate passed**: `npm test` passes with no failures.
- [ ] **Stakeholder-ready**: Labels, caveats, attribution, and trend context are present for public readers.
